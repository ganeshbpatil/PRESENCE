# Security Architecture

## Auth: Self-built JWT + OAuth (not Clerk — see stack decision record)
Rationale: avoids per-MAU vendor cost at target scale (8,000–10,000 businesses × multiple staff logins by Year 3), keeps auth data in the same Postgres instance rather than a third-party auth vendor holding customer identity data — relevant given DPDP Act data-residency/handling considerations for Indian SMB and (for the clinic segment specifically) health-adjacent customer data.

## RBAC Roles (v1 minimum)
- SMB Owner (full access to own business)
- Agency Admin (multi-client access, white-label console)
- Agency Viewer (read-only, for junior agency staff)

## Webhook Security (non-negotiable)
Every public-facing webhook (WhatsApp BSP, Meta, Razorpay) MUST verify signature/token before processing payload. This is enforced at the adapter level (`adapters/whatsapp/base.py:parse_webhook`), never trust-by-default. Traefik rate-limiting is a second layer, not a substitute for signature verification.

## Secrets Management
**Decided (2026-07-04): self-hosted HashiCorp Vault**, not Doppler/Infisical — no new vendor/billing, fits the existing VPS Pattern-A convention, keeps the "no AWS" constraint intact. Runs as a `vault` container on the isolated `presence-internal` docker network only (never on `traefik-public`, never internet-reachable) — see `docker-compose.yml` and `shared/secrets/vault_client.py`.

No raw platform access tokens (GBP/Meta/WhatsApp) are stored in application tables — `platform_connections.access_token_ref` is a KV v2 path reference (`platform-connections/<business_id>/<platform>`), never the raw token. `gateway/api/v1/businesses.py`'s connection-create endpoint takes the raw token as a write-only `access_token` field, writes it to Vault immediately via `VaultClient.store()`, and persists only the ref. `gateway/tasks/sync.py` resolves the ref back to a raw token via `VaultClient.resolve()` immediately before each Meta API call — the raw token is never persisted anywhere, only held in memory for the duration of that call.

Auth to Vault is AppRole (role_id + secret_id in `.env`, generated once by `scripts/vault-init.sh` — never committed, never generated by CI or an automated agent), scoped by policy to `presence-secrets/data/platform-connections/*` only.

Operational tradeoff, accepted deliberately: this is a single-node Vault with no cloud KMS available on this VPS for auto-unseal, so **Vault comes back sealed after every restart** and someone with the unseal keys must run `docker compose exec vault vault operator unseal` (x3) by hand. `gateway`/`celery-worker`/`celery-beat` do not hard-depend on Vault at boot — they resolve/store secrets lazily per request/task and fail closed with a clear error (502 on connection-create, `status="failed"` on a scheduled post) rather than blocking the rest of the stack from starting. `.env` remains local/dev-only as before.

## Data Protection (DPDP Act consideration)
Customer PII (phone numbers, review text, WhatsApp opt-ins) requires documented consent handling — build this into the onboarding/connect flow from v1, not retrofitted. Clinic-segment data (health-adjacent) warrants extra caution even though PRESENCE isn't processing clinical data directly — customer contact/visit data in a healthcare context still carries elevated sensitivity expectations.

## Payment Security
Razorpay handles PCI-scope directly (decision already made during LocalEdge exploration — do not re-evaluate). PRESENCE never touches raw card data.
